PHASE 01

VPC & Network Infrastructure

Pulumi provisions the entire network layer before any server is created. Everything lives inside a logically isolated VPC with two subnets — one public (internet-facing) and one private (DB-only).

VPC
Create the VPC — 10.0.0.0/16
FIRST ▶

A VPC is a private, isolated network inside AWS. The CIDR block 10.0.0.0/16 gives 65,536 IP addresses to work with. DNS support and hostnames are enabled so that EC2 instances can resolve each other by hostname rather than just IP.

__main__.py — VPC
vpc = aws.ec2.Vpc(
    'nodejs-db-vpc',
    cidr_block='10.0.0.0/16',       # 65,536 available IPs
    enable_dns_support=True,         # instances can resolve DNS
    enable_dns_hostnames=True,       # assigns .compute.internal hostnames
    tags={'Name': 'nodejs-db-vpc'}
)
CIDR 10.0.0.0/16
The /16 prefix means the first two octets (10.0) are fixed, giving 256×256 = 65,536 addresses for internal use.
DNS Hostnames
Each EC2 instance gets a hostname like ip-10-0-1-45.ap-southeast-1.compute.internal, useful for service discovery.
Isolation
Nothing outside the VPC can communicate with resources inside it unless you explicitly open a route (IGW, NAT, VPN).
SUB
Carve two subnets — public (10.0.1.0/24) & private (10.0.2.0/24)
NETWORK ▶

Subnets divide the VPC address space into smaller segments. The public subnet allows instances to get public IPs (map_public_ip_on_launch=True). The private subnet never assigns public IPs — instances there are unreachable from the internet directly.

Public subnet
public_subnet = aws.ec2.Subnet(
    'nodejs-public-subnet',
    vpc_id=vpc.id,
    cidr_block='10.0.1.0/24',          # 256 IPs for Node.js server
    map_public_ip_on_launch=True,      # auto-assign public IP to instances
    availability_zone='ap-southeast-1a'
)
Private subnet
private_subnet = aws.ec2.Subnet(
    'db-private-subnet',
    vpc_id=vpc.id,
    cidr_block='10.0.2.0/24',          # 256 IPs for DB server
    map_public_ip_on_launch=False,     # NO public IP ever
    availability_zone='ap-southeast-1a'
)

IGW
Internet Gateway — bidirectional door to the internet
ROUTING ▶

The Internet Gateway (IGW) is attached to the VPC and acts as the router between your VPC and the public internet. It performs NAT for instances with public IPs and routes external traffic in. Without it, nothing in the VPC can reach the internet — even with a public IP assigned.

Internet Gateway
internet_gateway = aws.ec2.InternetGateway(
    'nodejs-db-internet-gateway',
    vpc_id=vpc.id,   # attaches to your VPC
    tags={'Name': 'nodejs-db-internet-gateway'}
)

NAT
NAT Gateway + Elastic IP — one-way exit for private subnet
CRITICAL ▶

The NAT Gateway lives in the public subnet and has an Elastic IP attached. Private subnet instances (DB server) send all outbound traffic to NAT, which rewrites the source IP to the Elastic IP, forwards the packet through IGW, and translates responses back. The internet only ever sees the Elastic IP — never the DB's private IP. Crucially, the internet cannot initiate a connection to the DB server through NAT.

Elastic IP + NAT Gateway
# Step 1: Allocate a static public IP
elastic_ip = aws.ec2.Eip('nat-eip')

# Step 2: Create NAT Gateway IN THE PUBLIC SUBNET (critical!)
nat_gateway = aws.ec2.NatGateway(
    'nat-gateway',
    allocation_id=elastic_ip.id,   # bind the Elastic IP
    subnet_id=public_subnet.id,    # MUST be public — needs IGW access
    tags={'Name': 'nodejs-db-nat-gateway'}
)
⚠
Why public subnet? NAT itself needs to reach the internet to forward packets on behalf of the DB server. If you placed NAT in the private subnet, it would have no route to the internet — it would be a dead end. The public subnet has the IGW route, so NAT can get out.

RT
Route Tables — the traffic decision engine
ROUTING ▶

Route tables are the brains of subnet networking. Each subnet is associated with exactly one route table. When a packet leaves an instance, AWS checks the route table — longest prefix match wins. The 10.0.0.0/16 → local entry handles all intra-VPC traffic. The 0.0.0.0/0 catch-all sends everything else to either IGW or NAT depending on the subnet.

Public route table (→ IGW)
public_route_table = aws.ec2.RouteTable(
    'public-route-table',
    vpc_id=vpc.id,
    routes=[
        aws.ec2.RouteTableRouteArgs(
            cidr_block='0.0.0.0/0',         # all internet traffic
            gateway_id=internet_gateway.id,   # → Internet Gateway
        )
    ],
)
# + implicit: 10.0.0.0/16 → local (auto-added by AWS)

# Associate with public subnet
aws.ec2.RouteTableAssociation(
    'public-route-table-association',
    subnet_id=public_subnet.id,
    route_table_id=public_route_table.id
)
Private route table (→ NAT)
private_route_table = aws.ec2.RouteTable(
    'private-route-table',
    vpc_id=vpc.id,
    routes=[
        aws.ec2.RouteTableRouteArgs(
            cidr_block='0.0.0.0/0',          # all internet traffic
            nat_gateway_id=nat_gateway.id,    # → NAT Gateway (not IGW!)

PHASE 02

Security Groups — Firewall Rules

Security groups are stateful firewalls attached to EC2 instances. Each instance can have multiple groups. Inbound rules control who can connect; outbound rules control where the instance can reach. AWS automatically allows return traffic for established connections.

NSG
Node.js Security Group — public-facing ports
PUBLIC ▶

The Node.js server is intentionally internet-facing. Port 22 allows SSH from anywhere (in production you'd restrict this to your IP). Port 3000 is the application port. All outbound traffic is allowed so the server can install packages, call APIs, reach the DB, etc.

nodejs-security-group
nodejs_security_group = aws.ec2.SecurityGroup(
    'nodejs-security-group',
    vpc_id=vpc.id,
    ingress=[
        # SSH — allows you to log in and push scripts
        aws.ec2.SecurityGroupIngressArgs(
            protocol='tcp', from_port=22, to_port=22,
            cidr_blocks=['0.0.0.0/0'],   # ⚠ restrict to your IP in prod
        ),
        # App port — clients connect to Node.js app here
        aws.ec2.SecurityGroupIngressArgs(
            protocol='tcp', from_port=3000, to_port=3000,
            cidr_blocks=['0.0.0.0/0'],
        ),
    ],
    egress=[
        # All outbound allowed — npm, apt, DB connections
        aws.ec2.SecurityGroupEgressArgs(
            protocol='-1', from_port=0, to_port=0,
            cidr_blocks=['0.0.0.0/0'],
        )
    ],
)

DSG
DB Security Group — locked to public subnet only
PRIVATE ▶

The DB server's security group only accepts connections from the public subnet CIDR 10.0.1.0/24 — meaning only the Node.js server can reach it. No internet traffic, no other VPC, nothing else. Port 3306 (MySQL) and port 22 (SSH via ProxyJump) are both restricted to the same CIDR.

db-security-group
db_security_group = aws.ec2.SecurityGroup(
    'db-security-group',
    vpc_id=vpc.id,
    ingress=[
        # SSH only from public subnet — ProxyJump arrives from 10.0.1.x
        aws.ec2.SecurityGroupIngressArgs(
            protocol='tcp', from_port=22, to_port=22,
            cidr_blocks=[public_subnet.cidr_block],  # 10.0.1.0/24
        ),
        # MySQL only from public subnet — Node.js app connects here
        aws.ec2.SecurityGroupIngressArgs(
            protocol='tcp', from_port=3306, to_port=3306,
            cidr_blocks=[public_subnet.cidr_block],  # 10.0.1.0/24
        ),
    ],
    egress=[
        aws.ec2.SecurityGroupEgressArgs(
            protocol='-1', from_port=0, to_port=0,
            cidr_blocks=['0.0.0.0/0'],   # outbound for apt via NAT
        )
    ],
)
✓
Using public_subnet.cidr_block as the source is a Pulumi output reference — it dynamically resolves to 10.0.1.0/24 after the subnet is created. This keeps the config DRY and always in sync.
PHASE 03

EC2 Instances & SSH Configuration Injection

Two EC2 instances are created. After Pulumi resolves their IPs, it automatically writes an ~/.ssh/config file on your local machine — enabling one-command SSH access to both servers, including the private DB server via ProxyJump.

DB
DB EC2 instance — private subnet, no public IP
PRIVATE ▶

The DB server is provisioned first because the Node.js server's user-data script needs the DB's private IP baked into it. Pulumi creates this instance in the private subnet with no public IP. The key pair db-cluster must already exist in AWS (created with AWS CLI beforehand).

DB EC2 instance

    instance_type='t2.micro',
    ami='ami-01811d4912b4ccb26',    # Ubuntu 22.04 LTS in ap-southeast-1

NJS
Node.js EC2 instance — public subnet, user_data injection
BOOTSTRAP ▶

The Node.js EC2 instance gets a user_data script that runs automatically on first boot. This is the mechanism Pulumi uses to "push" configuration into the server without needing a separate SSH session. The script installs Node.js, writes the DB's private IP to /etc/environment, and drops the MySQL health-check script into /usr/local/bin/.

pulumi.Output.all()

user_data_replace_on_change

SSH
SSH config auto-injection via Pulumi Output callback
AUTOMATION ▶

After both instances are up and IPs are known, Pulumi calls create_config_file() which writes your local ~/.ssh/config. This file tells SSH how to reach each host — hostname, username, key file, and crucially the ProxyJump directive that makes DB access transparent.

SSH config generator (runs locally after pulumi up)
def create_config_file(all_ips):
    config_content = f"""
Host nodejs-server
    HostName {all_ips[0]}         # e.g. 54.179.120.45 (public IP)
    User ubuntu
    IdentityFile ~/.ssh/db-cluster.id_rsa

Host db-server
    ProxyJump nodejs-server        # tunnel SSH through nodejs-server
    HostName {all_ips[1]}         # e.g. 10.0.2.88 (private IP)
    User ubuntu
    IdentityFile ~/.ssh/db-cluster.id_rsa
"""
    config_path = os.path.expanduser("~/.ssh/config")
    with open(config_path, "w") as config_file:
        config_file.write(config_content)

# Wait for both IPs to resolve, then call the function
all_ips = [nodejs.public_ip, db.private_ip]
pulumi.Output.all(*all_ips).apply(create_config_file)

# Command you type:

# What SSH does internally:
# 1. Reads ~/.ssh/config, finds "Host db-server"
# 2. Sees ProxyJump=nodejs-server
# 3. Opens TCP connection to 54.179.120.45:22  (nodejs public IP)
# 4. Authenticates with db-cluster.id_rsa
# 5. Asks nodejs-server to open a TCP tunnel to 10.0.2.88:22
# 6. Runs a second SSH session THROUGH that tunnel to db-server
# 7. Authenticates again with the same key
# 8. You land in a shell on db-server
# — the key never leaves your machine (no agent forwarding needed)
PHASE 04

Script Push & Execution via User Data

The MySQL health-check script is read from disk by Pulumi, embedded into the user_data cloud-init script as a heredoc, and written to the Node.js server at boot time. No manual file transfers needed.

SH
The check-mysql.sh script — what it does
BASH ▶

This script runs on the Node.js server and checks whether MySQL port 3306 is reachable on the DB server. It uses netcat (nc) because it's lightweight — no MySQL client needed. It retries up to 30 times with 10-second pauses, so it handles the DB taking time to boot up.

script/check-mysql.sh
#!/bin/bash
DB_HOST="$DB_PRIVATE_IP"   # read from /etc/environment (set at boot)
DB_PORT=3306
MAX_RETRIES=30
RETRY_INTERVAL=10

check_mysql() {
    nc -z "$DB_HOST" "$DB_PORT"   # -z = scan only, no data sent
    return $?                      # returns 0 if port open, 1 if not
}

retry_count=0
while [ $retry_count -lt $MAX_RETRIES ]; do
    if check_mysql; then
        echo "Connected to MySQL at $DB_HOST:$DB_PORT"
        exit 0   # success
    fi
    echo "Attempt $((retry_count + 1))/$MAX_RETRIES: retrying..."
    sleep $RETRY_INTERVAL
    retry_count=$((retry_count + 1))
done

echo "Failed after $MAX_RETRIES attempts"
exit 1
nc -z (netcat zero-mode)
Opens a TCP connection to port 3306 and immediately closes it. Exit code 0 = port reachable, 1 = refused or timeout.
$DB_PRIVATE_IP
This environment variable is set at instance boot via /etc/environment by the user_data script — it's the real private IP of the DB server.
30 retries × 10s = 5 min
Gives MySQL up to 5 minutes to start up. Cloud instances and DB bootstrapping can take time — this prevents false failures.
INJ
How Pulumi embeds the script into user_data
HEREDOC ▶

Pulumi reads check-mysql.sh from disk into a Python string. Then the generate_nodejs_user_data() function builds a cloud-init bash script that embeds the check script using a heredoc (cat > file << 'EOL' ... EOL). This means the entire script is delivered to the server as part of the launch configuration — zero manual steps.

Reading script from disk
# Pulumi reads the script at plan time (on your local machine)

    mysql_check_script = file.read()  # stored as Python string
generate_nodejs_user_data() — full cloud-init script
def generate_nodejs_user_data(db_private_ip):
    return f'''#!/bin/bash
exec > >(tee /var/log/user-data.log) 2>&1
# ↑ logs everything to /var/log/user-data.log for debugging

apt-get update
apt-get upgrade -y
apt-get install -y netcat-openbsd   # needed by check-mysql.sh

# 2. Install Node.js 18.x
curl -fsSL https://deb.nodesource.com/setup_18.x | bash -

NC
Why netcat and not mysql client?
DESIGN ▶

Using netcat instead of the MySQL client is intentional. Netcat just tests TCP connectivity — it doesn't need credentials, doesn't install large packages, and works before MySQL is fully initialized. It answers the question: "Is the port open?" not "Can I authenticate?" — which is exactly what a readiness probe needs.

nc -z explained
# nc = netcat — the TCP/IP Swiss army knife
# -z = zero I/O mode (just scan, don't send data)
nc -z "$DB_HOST" "$DB_PORT"

# What happens under the hood:
# 1. nc opens a TCP socket to DB_HOST:3306
# 2. Completes TCP handshake (SYN → SYN-ACK → ACK)
# 3. Immediately closes connection
# 4. Returns 0 if handshake succeeded, 1 if refused

# Traffic path:
# nodejs (10.0.1.x) → 10.0.0.0/16 local route → db (10.0.2.x)
# No internet, no NAT — pure VPC-internal routing

SSH into DB server — final connectivity proof
END-TO-END ▶

With the SSH config written and MySQL reachable, the final step is SSH-ing directly to the DB server with a single command. ProxyJump handles the two-hop tunnel transparently. Once inside, you configure MySQL — install the server, set root password, create databases, and grant remote access to the Node.js server's private IP.

HOP 01
Your Laptop
runs the ssh command
reads ~/.ssh/config
You type: ssh db-server
SSH looks up Host db-server in your config. It finds ProxyJump nodejs-server — so before connecting to the DB it must first establish a connection to nodejs-server and tunnel through it. Your laptop never tries to reach 10.0.2.x directly.
# ~/.ssh/config — auto-written by Pulumi after `pulumi up` Host nodejs-server HostName 54.x.x.x # nodejs public IP (export from Pulumi) User ubuntu IdentityFile ~/.ssh/db-cluster.id_rsa Host db-server ProxyJump nodejs-server # ← this is the magic line HostName 10.0.2.x # db private IP (only reachable inside VPC) User ubuntu IdentityFile ~/.ssh/db-cluster.id_rsa
ℹ
Pulumi generates this file via pulumi.Output.all(nodejs.public_ip, db.private_ip).apply(create_config_file) — it only runs after both IPs are known, so the file always has real IPs.
HOP 02
nodejs-server
internet-facing
54.x.x.x:22 open
jump host
SSH connects to nodejs public IP — first leg
Your laptop opens a real TCP connection to 54.x.x.x:22. This works because the Node.js server has a public IP and its security group allows port 22 from anywhere. SSH authenticates using the private key. Once authenticated, SSH sends an internal protocol message asking the server to open a TCP channel to 10.0.2.x:22.
# What the nodejs security group allows — from __main__.py nodejs_security_group = aws.ec2.SecurityGroup( 'nodejs-security-group', ingress=[ aws.ec2.SecurityGroupIngressArgs( protocol='tcp', from_port=22, to_port=22, cidr_blocks=['0.0.0.0/0'], # port 22 open to internet ), aws.ec2.SecurityGroupIngressArgs( protocol='tcp', from_port=3000, to_port=3000, cidr_blocks=['0.0.0.0/0'], ), ], )
✓
Your laptop IP (203.x.x.x) hits port 22 → security group rule tcp:22 from 0.0.0.0/0 → ALLOW. First SSH session established.
Why is SSH open to 0.0.0.0/0? Because you need to reach it from your laptop which has an unpredictable IP. In production you would replace 0.0.0.0/0 with your own static IP — the Pulumi comment even says "Consider restricting to your IP". But functionally this is the public door into the entire system.
HOP 03
SSH Protocol
channel open
direct-tcpip
inside the tunnel
SSH asks nodejs to open TCP to 10.0.2.x:22
After the first SSH session to 54.x.x.x is open, your SSH client sends an SSH protocol message called channel open direct-tcpip. This is not a shell command — it is an SSH protocol feature. It tells the nodejs server's SSH daemon: "open a raw TCP socket to 10.0.2.x:22 and pipe everything through this channel." The nodejs server does this from its own network interface — which is inside the VPC and can reach the private subnet.
# You can see this happen with verbose SSH: ssh -vvv db-server # You will see output like: debug1: channel 0: new [direct-tcpip] debug1: Connecting to 10.0.2.x port 22 debug1: Connection established # SSH then runs a SECOND SSH session through channel 0: debug1: Authentication succeeded (publickey) debug1: channel 1: new [client-session]
ℹ
The key never leaves your laptop. The second SSH authentication (to the DB) also uses db-cluster.id_rsa from your laptop — SSH agent forwarding sends the crypto challenge back to your machine to sign.
HOP 04
DB security group
last firewall check
before the shell
DB security group checks the incoming connection
The TCP connection arriving at 10.0.2.x:22 comes from 10.0.1.x — the nodejs server's private IP inside the public subnet. The DB security group only allows port 22 from 10.0.1.0/24. Since 10.0.1.x is in that range, it passes. Notice that your laptop's IP 203.x.x.x never appears here — the DB sees the connection as coming from nodejs, not from you.
# DB security group — from __main__.py db_security_group = aws.ec2.SecurityGroup( 'db-security-group', ingress=[ aws.ec2.SecurityGroupIngressArgs( protocol='tcp', from_port=22, to_port=22, # ONLY from the public subnet — 10.0.1.0/24 cidr_blocks=[public_subnet.cidr_block], ), aws.ec2.SecurityGroupIngressArgs( protocol='tcp', from_port=3306, to_port=3306, # MySQL also only from public subnet cidr_blocks=[public_subnet.cidr_block], ), ], )
✓
Connection source = 10.0.1.x (nodejs private IP) → matches 10.0.1.0/24 → ALLOW port 22. DB accepts the tunnel.
✗
If someone tried ssh ubuntu@10.0.2.x directly from the internet — impossible. 10.0.2.x has no public IP and is not internet-routable. Even if they somehow got to the security group, 203.x.x.x does not match 10.0.1.0/24 → DENY.
HOP 05
Shell on DB
you are in
ubuntu@db-server
You land in a shell on the DB server
The second SSH session authenticates (same key, same user) and opens a shell. From your perspective you just typed one command. Under the hood two TCP connections were made and one was tunnelled inside the other. The DB server never needed a public IP — it only ever talked to the nodejs server on its private IP.
# Everything that happens from ONE command on your laptop: $ ssh db-server # SSH internally: # 1. open TCP → 54.x.x.x:22 (nodejs public IP) # 2. authenticate with db-cluster.id_rsa # 3. channel open direct-tcpip → 10.0.2.x:22 # 4. run second SSH session through channel # 5. authenticate again with same key Welcome to Ubuntu 22.04 LTS ubuntu@db-server:~$ # ← you are here # Verify you are on the private server: ip addr # shows 10.0.2.x — no public IP curl ifconfig.me # shows NAT Elastic IP (outbound only) hostname # db-server
★
Run sudo hostnamectl set-hostname db-server once inside to make the prompt show ubuntu@db-server instead of the AWS internal hostname. The Pulumi docs mention this step after first login.
WHY THIS DESIGN IS SECURE
✓
DB has no public IP — you cannot even attempt to connect to 10.0.2.x from the internet. It is not routable.
✓
DB SG allows port 22 only from 10.0.1.0/24 — even if someone got inside the VPC, they need to come from the public subnet.
✓
Key never leaves your laptop — ProxyJump does not copy your private key to the nodejs server. The crypto handshake comes back to you.
✓
Single key pair for both — db-cluster.id_rsa authenticates to both servers. Rotate one key and both are secured.
✗
Attacker can't bypass the jump — to reach 10.0.2.x:22 they must first compromise nodejs-server. There is no other path.
✗
DB has no way to be targeted directly — port scans from internet hit nothing. No open ports, no public IP, no route.
